# The opinionated guide to A+ Figma craft

A great Figma file is not a careful drawing. It is a **self-describing, structurally honest artifact** that any consumer — a designer browsing six months later, a developer in Dev Mode, a Code Connect mapping, a Figma Make prompt, an MCP-fed coding agent — can read without help. Everything in this guide pushes against the same enemy: pixel pushing that looks fine in the canvas and falls apart the moment anything changes. The features Figma shipped through Config 2024 and Config 2025 — Grid Auto Layout, Extended Collections, Code Connect, Make, Sites, the MCP server — collectively reward the same set of disciplines. **Variables instead of raw values, Auto Layout instead of free-form frames, components instead of duplicated shapes, semantic names instead of `Frame 23`.** Invest in those four habits and every new Figma surface you encounter rewards the work. Skip them and the file degrades into a static painting that fights you on every change.

This guide walks the full stack from the smallest layout primitive up through library architecture, with explicit anti-patterns called out as we go.

## Auto Layout is the default state of every UI frame

Auto Layout is Figma's flexbox-and-now-grid engine, and the consensus among Figma's own designer advocates (Luis Ouriach, Miggi Mendoza, Lauren Byrne, Joey Banks) is unambiguous: **wrap virtually every UI frame in Auto Layout by default**. The reason is structural, not stylistic — Auto Layout encodes the same intent the engineer will encode in CSS, so Dev Mode emits clean flex/gap output, Code Connect maps cleanly, Figma Make produces real React instead of `div` soup, and your design holds together when content changes length or locale. Joey Banks' rule of thumb is the right gut check: **drag an unfinished component and stretch it to bizarre dimensions; if it still works, you can trust it in production.**

The mental model that unlocks Auto Layout is the **Hug / Fill / Fixed** triple. Hug applies to an Auto Layout frame and means *shrink to children plus padding* — use it for buttons, chips, tags, tooltips, anything content-driven. Fill applies to a child and means *take all available space* — use it for input fields, table cells, the label in a row that should expand. Fixed locks a dimension — use it for icons, avatars, switches, anything whose size is invariant. The most common confusion: if any child is set to Fill, the parent can no longer Hug on that axis; it silently becomes Fixed. The keyboard tells: double-click an Auto Layout edge to Hug, Option/Alt + double-click to Fill. Multiple sibling children all set to Fill divide the available space equally, which is how you produce **equal-width columns without a grid**.

The 2023 additions — **min/max width and height, plus wrap** — are the difference between a layout that responds to content and one that quietly breaks. Min-width on a button keeps short labels from looking cramped; max-width on a paragraph caps reading line length without losing responsiveness. Wrap is available only on horizontal flow and is how you build tag clouds, filter chip rows, and responsive card grids that reflow from four-up to three-up to two-up as the parent shrinks. The single biggest reason designers find that "wrap doesn't work" is forgetting that **wrap requires explicit min-widths on the children to know when to break**. Min and max accept variables, so a single number token can drive responsiveness across the whole system.

**Absolute positioning inside Auto Layout** — renamed *Ignore auto layout* in UI3 — is the canonical solution to a recurring class of problems. Anchored notification badges, tooltips that should not push siblings, sticky bottom panels, floating action buttons, decorative overlays on hero images, scrollbars on top of menus, avatar stacks with custom z-order: all are children that escape the Auto Layout flow and pick up Constraints (top/right/bottom/left/center/scale) inside the AL parent. The drag shortcut to enable it is **Ctrl on Mac, S on Windows** while dragging into an AL frame. Negative gap (introduced 2022) handles overlapping stacks like avatar groups without needing absolute positioning at all; layer order in the panel sets z-order.

The **Grid auto-layout flow** added at Config 2025 is the right tool for true two-dimensional structures — bento grids, dashboards, product card galleries — and finally lets you express 1/3 + 2/3 splits without max-width hacks. It is not a replacement for the flex flows; one-dimensional content that needs to break to a new line still belongs in horizontal Wrap. Christine Vallaure's framing is correct: Figma's Grid is CSS Grid, so if you don't understand CSS Grid you'll fight it. Known limits as of 2026: no percentage tracks, no named areas, no subgrid, no Hug on rows or columns, no minmax(), and reordering layers only changes z-index, not display order.

A few non-obvious uses worth internalizing. **Auto-spacing dividers** are a single horizontal divider layer set to Fill width inside a vertical AL — the divider grows with the card. **Tooltip tails** are best handled as a component set with one variant per tail position, then placed via Ignore auto layout. **Equal flex columns** come from siblings all set to Fill, no manual width. **Variable-driven spacers** — a thin AL frame whose width or height is bound to a number variable — let you swap spacing values by mode. The "**space between**" navbar pattern is the single most-used AL recipe: horizontal flow, gap set to Auto by pressing X on the alignment 9-dot picker, the inner content automatically pushed to opposite ends, exactly mirroring CSS `justify-content: space-between`.

The named anti-patterns worth eradicating: **manual eyeballed gaps** instead of gap; fixed widths on children that should Fill; **spacer-frame DIV-itis** when a number variable on gap or padding would do; deeply nested unnecessary AL frames; fixed-size text boxes that overlap when content grows; sticky elements as regular AL children rather than Ignore auto layout; mixing groups and frames inconsistently. Dan Mall's heavy use of spacer frames is the lone defensible exception, and it's a personal style — the prevailing best practice is variables on gap.

When *not* to use Auto Layout: free-form illustrations, hero compositions where placement is by feel, marketing layouts with deliberately rotated and overlapping elements, FigJam diagrams, anything that lives in Figma Draw. The magic of Figma is moving fluidly between structure and free-form, but UI work belongs in structure.

## Constraints and layout guides earn a smaller, real role

Even with Auto Layout dominant, constraints have a clear job. They govern the children of non-AL frames, AL frames nested inside non-AL parents, and any child that ignores Auto Layout. That covers hero illustrations that need to scale with their frame, decorative shapes, marketing layouts, and every anchored badge or floating element living inside an AL parent. The **Scale** constraint is rarely correct for UI because text and icons distort, but it's the right choice for proportional artwork. Center constraints are how modals and loaders stay centered when the viewport resizes. Constraints don't work on groups — convert to a frame first.

**Layout guides** were renamed at Config 2025 to disambiguate them from the Grid auto-layout flow. They are visual overlays that don't move objects, but children with constraints can attach to a Stretch grid for legacy responsive flows still common in marketing pages. The 8pt grid system remains the de facto base unit, with 4pt for dense UI like toolbars and table rows. The relationship to Auto Layout: page-level grids for visual rhythm and alignment, Auto Layout for everything that contains components.

## Components — make fewer, expose more

The discipline that separates a working component library from a brittle one is restraint. A component should appear when **the same configuration shows up two or three times and you expect maintenance updates**. Pre-componentizing during exploration locks teams into wrong abstractions; Luis Ouriach's recommended path is to keep unfinalized work on a Discovery page, promote it to a *staging library* when it crosses team boundaries, and only graduate to the global library once stable. Carbon, Primer, Polaris, Material — every mature system maintains main components in dedicated published library files, never in product files where designers will instance from the wrong source.

The hardest decision in component design is **variant property versus separate component**. Alice Packard's flowchart, which has become the de facto reference, says: use variants when the structure is identical and only states change (size, hierarchy, count); split into separate components when there's a meaningful hierarchy difference (Primary may support split-button composition that Tertiary doesn't), when composition changes substantially across contexts, when the visual style is fundamentally different (filled versus stroked icons), or when independent versioning matters. The concrete case: button versus icon button. If padding and dimensions differ enough that an `iconOnly` boolean would warp the layout, ship `Button` and `IconButton` separately. If you can express it through a boolean and an Icon position variant cleanly, keep one component.

The threshold most teams hit is **variant explosion**. Figma's own button reportedly reached 1,600 variants before refactor. Steve Ruiz coined the right framing: combinatorial explosion doubles required states with each new parallel property. The decomposition tools are booleans for orthogonal toggles, instance swap for variable content slots, and separate components for hierarchy splits. One real case study cut a button from 160 variants to 80 by replacing a `withIcon=true/false` variant axis with a boolean property — saving nothing on capability but halving the maintenance surface.

**Never use variants for icons.** Figma's own help docs are explicit, and the reasons are structural: no preview in the variant dropdown, no search, all variants load to memory on a single instance, alphabetization is manual. Each icon should be its own component, with slash naming for organization (`Icon/16/check`, `Icon/24/check`, `Icon/Format/Bold`), a standardized frame size, a flattened inner shape, and a consistent inner layer name (typically `Vector`) so color overrides survive when swapping.

The five **component property types** each have a precise job. **Variant** properties handle mutually-exclusive enumerated states; they always pin to the top of the property panel and cannot be intermixed with other property types. **Boolean** properties bind only to layer visibility — they're the right tool for show-an-icon, has-divider, has-supporting-text. Phrase them positively so True reads as visible. **Instance swap** is for guarded single-item replacements with fixed footprint — icon slots, avatar slots — and you should always set Preferred values to scope the swap menu and a real default instance, not an empty placeholder. **Text** properties replace direct text overrides with a single field, valuable when the same string drives multiple nested elements (a label that also feeds an aria-label rendering). **Nested instance** properties — exposing a child component's properties up to the parent — are the power-user feature that flattens deeply nested input fields from sixty variants down to twelve. The known limitation is all-or-nothing: you cannot select which nested properties to surface, so keep nested components lean.

The **Slot** primitive moved into open beta in 2025 and is the first true composition primitive in Figma. Unlike instance swap, slots accept any layer types — text, frames, multiple items, images — and let the recipient resize and rearrange freely. The deliberate constraint is that you cannot bind component properties to layers inside a slot; that boundary is what preserves structural integrity. Use **preferred instances** as guardrails to suggest content. The decision rule is clean: variants for state, instance swap for guarded single replacement, slots for freeform composition.

Two architectural patterns matter beyond the property toolkit. **Compound components** — Nathan Curtis's framing — break a card into `Card`, `Card.Header`, `Card.Body`, `Card.Footer` components rather than one configurable mega-component. The trade is more discovery friction for richer, self-served composition that monolithic components never reach. **Layout primitives as components** — `Stack`, `Row`, `Spacer` — let designers compose layouts through the system rather than ad-hoc Auto Layout, with direction exposed as a variant.

Naming is what makes any of this discoverable. Component names go in **Title Case** (`Button`, not `btn`), with slash hierarchy producing the asset-panel folder tree (`Button/Primary/Large`). Property names are Title Case nouns (`Size`, `Type`, `Hierarchy`, `State`), values are lowercase if they map 1:1 to code props or Title Case if they're for designers — pick one and enforce. Boolean property values are stuck at True/False, so phrase the property name to make True read positively (`Show icon`, not `Hide icon`). Internal atomic components destined to be nested but not used directly are prefixed with `_` or `.` to exclude them from publishing. Component descriptions appear in the asset panel hover, in Dev Mode's inspect panel, and in instance Properties — treat them as keyword/alias fields, putting both the human name and the code identifier (`headerNavItemActive`) so cross-discipline search works.

A handful of named anti-patterns worth memorizing:

- **Variant explosion**, **booleans masquerading as variants** (or vice versa), **variants for icons**, and **mega-components** that resist subcomponent decomposition.
- **Detach-and-modify** instead of contributing back to the system, which produces silent design debt — Polaris is a notable cultural exception that explicitly permits detaching for exploration.
- **Naming inconsistency** that fractures the asset panel between `btn-primary`, `Button/Primary`, and `primaryButton` in one file.

Detached instances are detectable on Organization plans through Design System Analytics; the older Uber-team trick of planting a hidden "signature pixel" inside each component still works for plugin-based detection. Prevention is more architectural than tactical: provide slots and instance-swap properties so designers don't reach for detach in the first place.

## Variables, modes, and the three-tier token system

Variables are how a Figma file becomes a token system rather than a collection of magic numbers. The four primitive types — Color, Number, String, Boolean — cover the surface area of design decisions, but the architecture matters more than the mechanics. **The single most important pattern in modern Figma is the three-tier model**: a hidden Primitives collection holding raw values (`color/blue/500 = #0066FF`, `space/4 = 4`), a published Semantic collection expressing intent (`color/bg/surface`, `color/text/default`, `space/inset/md`) with modes for theming, and a hidden Component collection holding component-specific overrides (`button/primary/bg/default`) that always alias semantics, never primitives. Every mature system — Carbon, Material, Spectrum, Polaris, Primer, Atlassian — implements this same three-layer indirection. The payoff is real: re-aliasing the semantic tier swaps a brand wholesale; changing the brand blue means editing one primitive; component tunability never pollutes the semantic vocabulary.

The capability map of variables is worth knowing precisely because the gaps determine where styles still live. **Color variables** apply to fills, strokes, gradient stops (since 2024), shadow colors, and as the value inside color styles. Color variables cannot themselves be a gradient — variables hold one value, so multi-stop gradients remain in styles even when their stops are variable-bound. **Number variables** apply to corner radius, width and height, gap and padding, stroke weight, opacity, font size, font weight (numeric), line height in px (not %), letter spacing, paragraph spacing, and shadow X/Y/blur/spread. **String variables** apply to text content, font family (exact-match string), font style/weight name, and component variant property values used in prototyping. **Boolean variables** apply to layer visibility, component variant boolean properties, and prototype conditionals. Variable-font axes other than weight (slant, optical size, grade) cannot bind, line-height and letter-spacing accept px only, image fills cannot be variables, and effect styles themselves are not aliasable wholesale — you can save a shadow effect with all-variable properties, but "shadow variable" is not a thing.

**Modes** are how the same variable holds different values for different contexts. The canonical use case is Light/Dark, but the more interesting ones are density (comfortable/compact), brand for multi-brand systems, locale via string variables, device or breakpoint, accessibility modes for high-contrast and reduced-motion, and Alice Packard documented eleven atypical mode use cases including subscription tier, validation state, and seasonal theming. The mode limits jumped meaningfully on October 28, 2025: **Pro plans now support 10 modes per collection, Organization 20, Enterprise 40**, up from 4 across the board. Modes resolve hierarchically — set a frame's mode and descendants inherit unless they override, so a single canvas can hold a Light frame next to a Dark frame next to a Brand-B frame all using the same components.

**Scoping** is the highest-leverage discoverability tool in Figma and the most underused. Without it, a 2px border-radius primitive shows up when a designer is binding font size, padding, or X-offset. Scope `radius/sm` to `CORNER_RADIUS` and it only appears when binding corner radius. The plugin API enumerates every available scope: ALL_FILLS, FRAME_FILL, SHAPE_FILL, TEXT_FILL, STROKE_COLOR, EFFECT_COLOR for colors; TEXT_CONTENT, CORNER_RADIUS, WIDTH_HEIGHT, GAP, OPACITY, FONT_WEIGHT, FONT_SIZE, LINE_HEIGHT, LETTER_SPACING, PARAGRAPH_SPACING, PARAGRAPH_INDENT for numbers. **Hide from publishing** is the partner mechanism: prefix a collection name with `_` or `.` to hide the entire collection, or check Hide from publishing per variable to keep primitives private while only the semantic and component tokens reach consumers.

**Aliasing** is what creates the three-tier chain. A variable's value can point to another variable's value via curly-brace syntax (`{color/blue/500}`), and aliases survive mode switches by resolving through the *referenced* collection's current mode. Cross-collection aliasing is essential for multi-brand: each brand's Semantic collection aliases the same shared Primitives, but its `color/action/primary` resolves to a different primitive per brand. The component tier stays brand-agnostic because it references semantics.

Naming conventions deserve a dedicated paragraph because **flat or inconsistent variable names destroy a system faster than any other single mistake**. Slash naming is the Figma-native convention because forward slashes auto-create groups in the Variables modal — kebab-only collapses the hierarchy and prevents navigation. Nathan Curtis's taxonomy is the de facto reference: a token name layers `[namespace] / [object] / category / concept / property / variant / state / scale / mode`. The major systems each express this differently and looking at them side by side is clarifying. Material 3 uses three classes — `md.ref.*`, `md.sys.*`, `md.comp.*` — with paired `on-` foreground tokens for every surface (`md.sys.color.primary` and `md.sys.color.on-primary`). Carbon v11 uses descriptive layered tokens (`$layer-01`, `$field-01`, `$button-primary-active`) after renaming v10's numbered `ui-01` set. Polaris uses kebab with a `--p-` prefix and a 4px-based numeric scale (`--p-space-100` = 4px, `--p-space-400` = 16px). Primer splits into Base, Functional, and Component tiers (`base-color-green-5`, `bgColor-inset`, `button-primary-bgColor-hover`). Spectrum uses Global, Alias, Component-specific tiers with the `--spectrum-*` prefix. The patterns differ; the discipline of three tiers and consistent anatomy is universal.

The named anti-patterns in token naming: too-flat names (`bg-primary`, `text-color`); color-naming the semantic tier (`color-text-red` breaks the moment error stops being red — use `color/text/error`); component tokens that reference primitives directly, collapsing the indirection that makes theming work; sequential numeric suffixes that imply ordering you'll later regret (use 100-step scales so you can insert a step); state position drift across tokens; and hardcoded specificity baked into shared tokens (`card-customer-page-bg` belongs as a frame override, not in the system).

The **W3C Design Tokens Format** (DTCG) reached its first stable spec on October 28, 2025, standardizing the JSON shape — `{$value, $type, $description}`, curly-brace aliases, twelve standard types from `color` and `dimension` to `transition` and `gradient`. Figma's plugin and REST APIs align conceptually but Figma still delegates DTCG import/export to plugins, primarily Tokens Studio.

For multi-brand specifically, **Extended Collections** rolling out from Schema 2025 (Enterprise, November 2025) is the new recommended pattern. A parent collection holds canonical structure and default values; child collections inherit and override only what differs, with overrides shown by a blue indicator and non-overridden values staying live-linked. This finally matches how dev-side theme inheritance has worked all along (CSS variable cascade, theme JSON inheritance) and replaces older swap-library and duplicate-and-fork patterns. Pre-Extended-Collections, the most common multi-brand pattern was either mode-based theming in one Semantic collection or separate published libraries per brand.

**Conditional logic and expressions** in Figma are a 2024 addition that lives inside *prototyping*, not inside variable definitions. The `Set variable → to:` field and the `Conditional → if:` field accept arithmetic (`+ - * / %`, parens), comparison (`== != < <= > >=`), logical operators (`&& || !`), and string concatenation. You can write `cartTotal = cartTotal + itemPrice` on a click and `if cartTotal > 50 → Set variable freeShipping = true`. The key constraint is that **variables themselves cannot store a formula**. A native `space/lg` cannot be defined as `{space/sm} * 2` — that arithmetic only works inside Tokens Studio's plugin layer, where it resolves before publishing. Late 2025 commentary suggests computed variable values are moving toward GA; treat as preview until release notes confirm.

## Where styles still live, and what they're for

With color variables shipping, **color styles are largely superseded** and exist mainly for backward compatibility and for layered or multi-fill compositions. The interesting question is what styles are still uniquely useful for, and the answer is anything that bundles multiple values into one decision. **Text styles remain required** because they're the only way to package family + weight + size + line-height + letter-spacing + paragraph spacing into one applicable atom. Each of those properties can be variable-bound inside the style, so a single text style can shift across desktop/mobile via number variable modes or across English/Japanese via family string variables — but the style itself is the carrier. **Effect styles remain required for shadows.** Inside an effect style, X/Y/blur/spread accept number variables and color accepts a color variable, but the effect itself cannot be a variable, and multi-shadow stacks must live as styles. **Gradient styles remain required for multi-stop gradients**, even though individual stops are variable-bindable. **Image fills** can only be styles; there is no image variable.

The migration path from styles to variables is a one-time investment and worth doing cleanly. Audit existing color styles in Selection Colors view, build a hidden Primitives collection from raw hexes, build a Semantic collection with Light/Dark modes aliasing primitives, then for each existing color style edit its fill to bind to the appropriate semantic variable — this preserves all instance bindings. Then deprecate the style. Schema 2025 introduced **Check Designs** in early access for Enterprise, which auto-suggests the right variable for any hard-coded value and makes auditing tractable.

## Library architecture and the system of systems

The dominant 2025 pattern, championed by Nathan Curtis and Luis Ouriach, is a **system of systems** rather than a monolithic library. Curtis names three tiers explicitly: core libraries owned by the design system team (tokens, primitive components, icons), shared/contributor libraries built by extension teams (charting, marketing patterns), and local/staging libraries that are team-specific. The mature split most large systems converge on is Foundations/Tokens, Icons (kept separate so the asset panel doesn't drown), Core Components, Patterns, optional Platform-specific libraries (Mobile/Web/Desktop) layered on top, and Brand/Illustration libraries.

The triggers for splitting are concrete: file performance and memory limits, divergent ownership, divergent publishing cadence (foundations change rarely, patterns change weekly), and audience subset (mobile-only teams shouldn't see desktop components). Figma's own best-practice doc endorses splitting so libraries can be enabled selectively per team. The counter-trend worth noting: some practitioners are reverting from heavy splits to fewer larger libraries combined with branching, finding that split libraries multiply maintenance overhead. The honest answer depends on team size, performance ceiling, and ownership boundaries — there is legitimate disagreement here.

Cross-library dependencies create real risks. **Dependency cycles** — a Components file pulling from Tokens, Patterns from Components — are fine; never let Components reference Patterns or you produce non-deterministic publish ordering. **Broken references** when a downstream-used token is renamed force every consuming library to republish; Curtis's prescription is additive-only deprecation with a `[Deprecated]` prefix and a replacement note rather than rename-in-place. **Library overflow** — Curtis's observation that the libraries dialog now overflows with tens or hundreds of options — argues for aggressive curation and Org/Enterprise library scoping that publishes only to relevant teams.

Publishing discipline matters more than people realize. Bundle changes thematically rather than piecemeal, **always fill in the description of changes**, because only the first user to accept gets the auto-generated diff list — every subsequent designer sees only your description, and after acceptance the diff vanishes. Maintain a manual `🗒 Release Notes` page in the library file itself with semantic versions, exactly as Polaris does in their UI kit. Use staged publishing on Org/Enterprise — publish first to the design-system team scope, validate against test files, then publish to the whole organization. Never publish at end-of-day Friday.

**Branching** is the right tool for system updates, large refactors spanning many components, and exploratory sandboxes — and the wrong tool for one-off tweaks or daily product design. Polaris mandates a branch per component change, named with the GitHub issue number (`[4963] Navigation design changes`). Phoenix's Compound design system caps branch lifetime explicitly: branches should not remain open longer than necessary, and accumulated drift past a month produces conflict storms. **Pull from main daily**, keep branches short-lived, update existing components rather than delete-and-recreate (which loses instance links), require two approvals before merge, and write a meaningful merge description because it becomes a permanent version-history entry. Figma's merge is binary — Keep main / Keep branch — so first-to-merge wins; coordinate on Slack rather than relying on Figma to mediate.

Versioning conventions inside Figma converge on a small set of habits. Use named versions (right-click in version history) as semantic anchors — `v2.4.0`, `v2.5.0-beta` — set before any major publish. Semantic versioning maps to Figma changes naturally: MAJOR for breaking renames or removals, MINOR for additive components or variants, PATCH for visual tweaks. **File-name versioning (`Components v3`) is now an anti-pattern** outside intentional generational forks; rely on version history and named versions instead.

## File structure that reads itself

A library file's page structure should be legible at a glance. The 2025 canonical structure: a 🎨 Cover page first (the thumbnail), 📖 Getting Started or About, 🗒 Release Notes, 🎨 Foundations or Tokens for visual reference (the variables themselves live in the Variables panel), 🔣 Icons (or a separate library if very large), 🧩 Components (one page per category for medium libraries, one page per component family for very large systems like Material 3), 🧱 Patterns for composed reference layouts, 🏖 Sandbox for WIP, 🗑 Archive for deprecated work, and `---` spacer pages between phases. Hidden internal pages are prefixed with `_` or `.`. Cover is always first because it provides the file thumbnail.

**Sections versus frames versus groups** is one of the cleanest decision rules in Figma. Frames are the workhorse for everything that contains other layers — UI canvases, components, cards, anything that needs Auto Layout, constraints, prototyping, clipping, fills, exports. Sections, introduced 2023, are top-level canvas organization for grouping multiple frames into a feature flow or contribution; their primary 2026 purpose is that **Mark as Ready for Dev applies cleanly at the section level**, making sections the standard handoff unit. Sections cannot live inside frames or components. **Groups should be almost-never** — UI Prep's challenge is the right one: only use frames. The narrow defensible case is vector art symbols where bounds should auto-fit. Pasting from external sources (Sketch, Illustrator, screenshots) creates groups; convert immediately.

**Cover thumbnails** at 1920×1080 should communicate at a glance: file name, owner, status (WIP / In Review / Ready / Deprecated), version, last update. A status indicator color system (red WIP, amber review, green ready, gray archived) on the cover plus dev-mode statuses on sections is more informative than encoding status in page-name emoji — Alice Packard's critique is that status-emoji churn produces spurious "unpublished changes" because page names are part of library metadata. Emoji for *purpose* is fine; emoji for *status* belongs in Dev Mode.

Layer naming is where most files fail and where the most leverage hides. **Auto-generated names like `Frame 1234`, `Rectangle 47`, `Group 12` are an anti-pattern**, and the consequences compound across every Figma surface. Dev Mode surfaces layer names as element labels. Code Connect maps Figma component and property names to code identifiers. Smart Animate matches layers between source and destination frames *by layer name and hierarchy position* — sloppy names break previously-working animations silently. Asset panel search relies on names. Buzz templates expose layer names as Edit-content field labels to non-designer end users. Exports generate filenames from layer names. The MCP server feeds these names to AI agents as context. The cost of `Frame 23` is no longer one developer's frustration; it's a quality regression across at least six product surfaces. Cmd/Ctrl + R supports bulk rename with regex.

The disciplined cleanup pass after pasting from any external source: run **Clean Document** (delete hidden layers, ungroup single-layer groups, pixel-align, intelligently rename), convert all groups to frames, apply Auto Layout where structure repeats, replace raw colors with variables, replace text formatting with text styles, then rename meaningful layers. Other useful tools in the same family are Design Lint for token and consistency issues, and Clean Layers for unnamed and untethered components. Flatten decorative vectors once finalized for performance; **never flatten functional UI**, which destroys Auto Layout and prototyping links.

## Prototyping with variables retires most prototypes

Variables in prototypes are the single biggest prototyping shift since interactive components. The **Set variable** action on any trigger lets you change a string, number, boolean, or color variable's value, and binding variables to layer properties (text content, dimensions, **layer visibility**, fills, stroke, opacity) lets you build state-driven prototypes with far fewer frames. A login form that previously required a frame per field state collapses to a single frame whose visibility, error text, and button enabled state are all variable-driven.

**Conditionals** evaluate boolean expressions and trigger different actions for if/else branches; stacking conditions handles multi-branch logic, and order matters since actions execute top-to-bottom. **Expressions** support the operator set described above. The honest practitioner caveats: no arrays, no functions, no loops, no clean way to do dynamic math without intermediate variables. For mutually-exclusive selection like radio buttons or active nav highlighting, use a single string variable holding the active item's ID rather than multiple booleans — it scales and stays consistent.

**Variable modes in prototypes** let Set variable target a specific mode value, which is how you build dynamic light/dark toggles in a single frame.

**Smart Animate** is where naming earns its keep most visibly. The match algorithm requires layer name match (case-sensitive), hierarchy match (same parent depth), and layer type match — a rectangle cannot smart-animate into a text node. Top-level frame names don't need to match; only the children. For text layers, if implicitly named (auto-named from content), text styles must match across frames. Within a section, matches are sought only inside that section. **Cmd+D to duplicate frames is preferred over copy-paste from elsewhere** because it preserves naming. The implication compounds: AI Rename Layers run sloppily can break previously-working animations.

## Dev Mode prep is file craft, not handoff communication

The file-side prep for Dev Mode is a checklist, not a process. **Wrap every shippable unit in a Section, named after the feature, and apply Ready for dev only when content is locked.** WIP and exploration belong on a separate page so the Ready-for-dev view stays clean; any content not in a section is collapsed by default in Dev Mode, so sections are the focusing mechanism. Org/Enterprise plans add a Completed status that closes the loop, and a yellow alert flags drift if a designer edits after marking ready.

**Annotations** (Shift+T for text, Shift+M for measurements) are the right tool for non-visual requirements — accessibility (ARIA labels, focus order, keyboard shortcuts), interaction behavior (debounce, polling), conditional logic, internationalization, analytics events, API field mapping. The +Property chip binds an annotation to a specific layer property so it auto-updates with design changes — eliminating stale callouts. Categories are file-scoped and filterable; consistent category use is what makes the filter useful. Anything self-evident from the design (padding, color, font) does not belong in an annotation; Dev Mode shows it natively. Discussion belongs in comments, not annotations. Long-form documentation belongs at a link, attached via Dev Resources on the component. The known constraint: annotations are visible only with Full or Dev seats, so non-paying viewers can't see them.

**Code Connect** went GA at Config 2024 and the file-side implications are precise. Each Figma component should map cleanly to one code component; if your code has three separate `PrimaryButton`, `SecondaryButton`, `DangerButton` components but Figma has one Button with a Type variant, use variant restrictions in `figma.connect()` to map each variant to the appropriate code component. **Align Figma property names with code prop names** — Figma `disabled` boolean → code `disabled` prop. Use variants for what code expresses as enums, booleans for `disabled`/`loading`, instance swap for slottable children that map to JSX `children`. The 2025 Code Connect UI added in-Figma point-and-click mapping with GitHub integration and is what powers the Figma MCP server's component context. Custom MCP instructions per component let you inject accessibility rules and prop patterns into the AI agent's context.

## The new AI surface area rewards the same disciplines

Figma Make, announced at Config 2025, is a prompt-to-code product that generates functional React/HTML/CSS apps from prompts and pasted Figma frames. The fidelity of its output is almost entirely determined by the file you give it. **Auto Layout is critical** because Make uses AL hierarchy to infer flex/grid containers, gap, padding — free-positioned frames become absolute-positioned div soup. Real components and instances let Make recognize repeated structures and produce reusable React components. Variables get picked up as tokens. Layer naming flows through to the generated code. **Make kits**, shipped late 2025, are the quality lever — a kit bundles an npm package containing your real React components, imports your Figma variables and styles, and includes Markdown guidelines files written in imperative voice (`Do not use small text except for captions`). With a kit attached, Make outputs use your actual production components rather than generic Tailwind-ish HTML. Without one, treat Make output as throwaway exploration.

**Figma Sites**, also Config 2025, lets you publish responsive websites directly. Auto Layout is mandatory for responsive behavior. Each page starts with a Desktop primary breakpoint; edits cascade to secondary breakpoints (Tablet, Mobile, custom widths) but can be overridden per-breakpoint. The most powerful pattern is **responsive components** — a component set whose Variant property values match breakpoint names, which Sites auto-swaps when the instance enters a matching range. Hover-based interactions don't fire on touch, so swap to press effects at mobile. Set `nav`, `main`, `footer` accessibility landmarks on top-level frames because they affect published HTML and SEO. Mixing Group with Frame nesting produces unpredictable responsive behavior; fixed-width top-level sections break on mobile.

**Figma Buzz** is the marketing/template product. Templates need Auto Layout on every layer so swapped images and long headlines don't break alignment. Locked layers don't appear in the end-user Edit content panel, which is the focused-editing mechanism — though end users must open a *new file from a published template* for lock restrictions to apply. **Name every editable layer semantically** because end users see those names as field labels in Edit content. Component sets become Template Sets; boolean props become toggle switches; instance swap becomes dropdowns; text properties become Edit content fields. Variable modes drive multi-theme and multi-locale templates. CSV/XLSX bulk create maps columns to layer names — naming clarity directly affects mapping clarity.

**Figma Slides**, GA from Config 2024, lets you use Figma Design libraries directly — components, styles, variables all carry over. Live interactions (Polls, Stamps, Likert scales, embedded prototypes) are first-class. Variable modes apply per slide. Treat Slides files like design files: components, styles, and variables for templated layouts.

The **AI features** in Figma Design proper are useful but require discipline. First Draft generates wireframes from a UI Kit and a prompt — treat output as wireframe, not pixel-perfect, and apply your real design system afterward. Make Image generates raster images using ChatGPT Images 2.0 or Google Imagen 3 and is for placeholder only — production usage raises licensing and brand issues. **Rename Layers** is content-aware AI; honors already-renamed layers, skips component instance sublayers and primitives, and is best run *after* the design is structurally final so the AI sees full context. Smart Duplicate generates content variations for list items and dashboards — replace before handoff. Visual Search is the under-rated one because it lets you find an icon by drawing it rather than by remembering whether it was named `chevron-right` or `arrow_right_16`. The repeating anti-pattern: accepting AI output unchanged, then not reviewing renamed layers and breaking Smart Animate downstream.

The **Figma MCP server** (beta, 2025) bridges Figma context to AI agents like Cursor, Claude Code, Codex, VS Code, and Windsurf. Agents read variables, components, and layer structure directly. **File craft for MCP is the same file craft that has been right all along**: variables, Auto Layout, components, semantic naming, Code Connect mappings. The headline observation across all the new products: the same well-crafted Figma file now feeds three different AI/code surfaces — Dev Mode plus Code Connect, Make, and MCP-driven IDE agents. Sloppy files used to frustrate developers; they now produce bad output across all three.

## A consolidated catalog of what to never do

Across the surfaces above, the same anti-patterns recur. Build the muscle to spot them:

- **Manual eyeballed gaps and fixed widths where Fill belongs**, fixed-size text boxes that overlap when content grows, deeply nested unnecessary Auto Layout frames, sticky elements as regular AL children rather than Ignore auto layout, and the spacer-frame DIV-itis when a number variable on gap or padding would do.
- **Variant explosion**, booleans masquerading as variants and vice versa, variants for icons, mega-components that resist subcomponent decomposition, detach-and-modify instead of contributing back, components created too early during exploration, and naming inconsistency that fractures the asset panel.

Beyond those, the persistent file-level issues: hardcoded hexes and magic-number spacing instead of variables; flat or kebab-only variable names that destroy hierarchy; color-naming the semantic tier; component tokens referencing primitives directly; auto-generated layer names like `Frame 23` and `Rectangle 47`; stray groups from external pastes; hidden layers left invisible rather than deleted; single-layer groups that are always redundant; file-name versioning instead of named versions; publishing without filling in the change description; long-running branches that accumulate conflict storms; status emoji on page names that produce spurious unpublished-changes noise; pasting raw images for every Slides slide instead of templated components; and Group/Frame mixing in Sites that breaks responsive behavior.

## What changes when the file is the artifact

The thread connecting every section of this guide is that a Figma file is increasingly read by software, not just by humans. Dev Mode reads it. Code Connect reads it. Make reads it. The MCP server reads it on behalf of an IDE-resident coding agent. Buzz reads it on behalf of a marketer. Sites reads it as a publishing source. Slides reads it as a presentation library. Each of those readers rewards the same six habits — variables over raw values, Auto Layout over free-form frames, components over duplicated shapes, sections to scope shippable work, semantic layer naming, and Figma property names that match code prop names — and punishes their absence in compounding, increasingly visible ways.

The shift worth internalizing is that **"clean" is no longer a stylistic choice or a craft brag**. A clean Figma file in 2026 is the structural prerequisite for everything downstream: theme switching across modes, multi-brand inheritance through Extended Collections, generated React from Make, IDE-side AI generation through MCP, automated marketing assets from Buzz, responsive websites from Sites, and dev handoff that doesn't require translation. The investment compounds because every new surface Figma ships keeps cashing in on the same disciplines.

The corollary is that what used to feel like over-engineering — a hidden Primitives collection, scoped variables, a three-tier token chain, named components with code-aligned property names, sections marked Ready for dev — now is the baseline. **Pixel pushing is no longer cheap**, because every shortcut taken in the file degrades not one consumer but six. Build the file for the consumer that will read it most ruthlessly, which from now on is increasingly a model.
